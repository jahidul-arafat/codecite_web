FAQ for Reviewers

Addressing common technical questions from peer reviewers

Q1Why is Œ±=0.3 the optimal sparse weight?

Key Finding: Code comprehension queries rely more on semantic similarity than exact keyword matching.

  • Œ±=0.3 (sparse): Captures exact function/variable names
  • Œ≤=0.7 (dense): Understands conceptual queries like "how does routing work"
Grid search across Œ±‚àà{0.1, 0.3, 0.5, 0.7} showed 0.3 maximizes citation accuracy by +23% over sparse-only.
Q2Why use BM25 + CodeBERT instead of just dense?

Complementary Strengths:

  • BM25: Exact matches for dispatch_request, url_for
  • CodeBERT: Semantic understanding of "error handling" ‚Üí handle_exception
Hybrid achieves 84.3% accuracy vs 71.2% (sparse) and 76.5% (dense).
Q3What is RM3 query expansion and why +18% recall?

Mechanism: Pseudo-relevance feedback extracts terms from top-10 initial results.

Original Query:
"Flask routing"
Expanded:
"Flask routing dispatch url_rule endpoint redirect"
Q4Cross-encoder reranking adds +12% but 2√ó latency‚Äîworth it?

Trade-off Analysis:

  • Pro: Joint query-document scoring improves precision significantly
  • Con: 2√ó latency (2.8s ‚Üí 3.8s average)
Recommended for accuracy-critical applications; optional --no-rerank for speed.
Q5How does FAISS HNSW enable fast dense search?

HNSW (Hierarchical Navigable Small World): Graph-based approximate nearest neighbor.

  • O(log n) query time vs O(n) brute force
  • 768-dim CodeBERT vectors indexed
  • Billion-scale capability
Q1Why Neo4j instead of SQL for code graphs?

Complexity Comparison:

  • Neo4j: MATCH (a)-[:IMPORTS*1..2]->(b) ‚Äî O(k) traversal
  • SQL: Multiple self-joins ‚Äî O(k√ón) complexity
Code dependency graphs are naturally hierarchical; graph DBs are 10-100√ó faster for hop queries.
Q2Why 2 hops optimal? Why not 1 or 3?

Empirical Finding:

  • 1-hop: Misses transitive dependencies (e.g., routing ‚Üí app ‚Üí config)
  • 2-hop: Sweet spot‚Äîcaptures cross-file context
  • 3-hop: Introduces noise, dilutes relevance
2-hop: 89.1% coverage | 3-hop: 87.8% (noise penalty)
Q3What is the decay factor (Œ≥=0.6)?

Purpose: Attenuate score contribution as hop distance increases.

S_g(d) = S(q,d) + 0.25 √ó 0.6^h

Hop 1 bonus: 0.15 | Hop 2 bonus: 0.09

Q4Does CodeCite work without Neo4j?

Yes, with degradation:

  • Use --no-graph flag
  • Cross-file coverage drops from 89.1% ‚Üí 58.2%
  • Citation accuracy: 92.4% ‚Üí 84.3%
Q5What edges are captured in the code graph?
  • imports: Module dependencies
  • calls: Function invocations
  • extends: Class inheritance
  • uses: Variable/attribute access
Q1Why submodular packing beats greedy by +15%?

Key Difference:

  • Greedy: Selects top-k by score‚Äîoften redundant chunks
  • Submodular: Maximizes diversity: F(S) = Œ£rel + Œª¬∑cov - Œº¬∑red
Submodular ensures coverage across different files/modules, not just highest-scoring duplicates.
Q2What are Œª and Œº in the objective function?

Hyperparameters:

  • Œª=0.4: Coverage bonus (distinct files/modules)
  • Œº=0.2: Redundancy penalty (pairwise similarity)
Q3Why 12K character budget?

Context Window Optimization:

  • 8B LLMs: ~4K-8K tokens effective context
  • 12K chars ‚âà 3K tokens + room for prompt + response
  • Larger budgets showed diminishing returns
Q4How many chunks typically selected?

Typical Output: 6-10 chunks from 100 candidates

Average chunk: 60-80 lines √ó ~20 chars/line = ~1.2-1.6K chars

Q5What is the (1-1/e) approximation guarantee?

Theoretical Bound: Greedy forward selection for monotone submodular functions achieves at least 63% of optimal.

Practical performance often exceeds this bound.
